# Megatron-LM 学习策略（学习管线）

本文档约定：**按「功能 → 原理 → 实现 → 代码逻辑与调用过程」** 的顺序系统学习本仓库，并与官方文档对照。

---

## 1. 总体策略

- **一条主轴**：先弄清「一次训练 step 里发生了什么」，再拆成 **数据 / 前向 / 反向 / 通信 / 优化器 / checkpoint**，最后把每条线接到 **并行策略** 上。
- **两条阅读线**（建议穿插，不必严格先后）：
  - **官方文档**（概念与术语）：仓库 README 中的 [Parallelism Strategies](https://docs.nvidia.com/megatron-core/developer-guide/latest/user-guide/parallelism-guide.html)、[Quickstart](https://docs.nvidia.com/megatron-core/developer-guide/latest/get-started/quickstart.html)。
  - **本仓库代码**（与实现对齐）：`megatron/core/` 为核心库；`megatron/training/` 为如何把模块拼成可跑的训练流程。

### 1.1 仓库分工（心智模型）

| 组件 | 定位 |
|------|------|
| **Megatron Core** (`megatron/core/`) | 可组合的 GPU 优化构件：Transformer、并行、精度、部分推理与导出等。 |
| **Megatron-LM**（本仓库其余部分） | 在 Core 之上的参考训练脚本、示例与工具，适合学习与实验。 |

## 2. 每个主题的笔记模板（统一复用）

每学完一小块，按下面四栏整理（可写在 `study/notes/` 等子目录）：

1. **功能**：解决什么问题、谁在用、边界是什么。
2. **原理**：关键不变量（例如张量如何切分、梯度如何聚合、通信模式）。
3. **实现**：对应包 / 类 / 文件路径（Megatron 里通常落在 `megatron/core/...`）。
4. **调用过程**：从入口到该点的调用链（函数名可逐步补全）：初始化并行 → 建图 → `forward` → `loss` → `backward` → 梯度处理 → `optimizer.step`。

## 4. Phase 1：单卡或最小分布式上的「训练一步」（约 2～4 天）

**目标**：暂不深入并行细节，先画出 **一次 iteration 的数据流**。

| 顺序 | 主题 | 学习顺序（功能→原理→实现→调用） |
|------|------|----------------------------------|
| 1 | 配置与模型构建 | 超参 → `TransformerConfig` / Module；spec 与模块化；训练脚本里 build model |
| 2 | 数据与 batch | micro-batch、global batch；sample 如何映射到 rank；`megatron/core/datasets/` 与 training 侧 dataloader |
| 3 | 前向：Transformer 块 | 单层 attention / MLP 做什么；形状与激活；`megatron/core/transformer/`（如 `attention.py`、`mlp.py`） |
| 4 | 损失与反向 | loss 在哪算；autograd；训练循环中 loss 与 `backward` |
| 5 | 优化器与精度 | Adam、混合精度等；`megatron/core/optimizer/` |

**阶段自检**：能按顺序口述从「拿到 batch」到 `optimizer.step」的主要环节（函数名可先模糊）。

## 5. Phase 2：并行策略（约 1～2 周，按依赖关系）

对每一种并行，都严格走 **第 2 节** 的四栏模板。

**建议顺序**：**数据并行（DP / 相关分布式封装）→ 张量并行（TP）→ 流水线并行（PP）→（按需）上下文并行（CP）→ MoE / EP**。

| 策略 | 功能（一句话） | 原理要点 | 实现落点（仓库） | 调用过程关注点 |
|------|----------------|----------|------------------|----------------|
| DP / DDP / FSDP 等 | 多卡复制或分片参数，同步梯度或分片优化器状态 | all-reduce、reduce-scatter、分片 | `megatron/core/distributed/` | grad finalize、与 optimizer 的衔接 |
| TP | 层内切分，降低单卡参数与激活 | 列/行切分、matmul/attention 处通信 | `megatron/core/tensor_parallel/` | `all_gather` / `reduce_scatter` 在何处触发 |
| PP | 层间切分，用调度换显存 | micro-batch、schedule、激活在 stage 间传递 | `megatron/core/pipeline_parallel/` | forward-backward 调度、bubble |
| CP | 长序列维切分 | 序列维上的集合通信与内核配合 | Core 内 context parallel 相关模块（含 SSM 等场景） | 与 attention 形状、通信的配合 |
| EP（MoE） | 专家并行 | token 路由、all-to-all 等 | `megatron/core/transformer/moe/` 等 | router + dispatch / combine |

**每学完一种**：画「本 rank 张量形状变化 + 通信原语」示意图，并对照 wrapper / communicator 代码。

---

## 6. Phase 3：系统能力（与 Phase 2 交叉、按需）

- **Checkpoint / 分布式检查点**：功能（恢复与续训）、原理（分片格式）、实现（`megatron/core/dist_checkpointing/`）、调用（save/load 在训练循环中的触发点）。
- **Inference**：与训练不同的 batching 与 KV；`megatron/core/inference/`。
- **RL / post-training**：在 Phase 1～2 清晰后再接（例如 `train_rl.py`、`megatron/rl/`），避免被多进程 rollout 干扰主线。

---

## 7. Phase 4：精读一条真实调用栈（约 3～5 天）

选定 **一个** `examples/` 脚本（或等价入口），从入口跟到：

1. 解析参数 → 并行初始化（process group、parallel state）。
2. 构建 model（是否包 TP/PP 等 wrapper）。
3. 训练循环：`forward` → `loss` → `backward` → **梯度处理（finalize / overlap）** → `optimizer.step`。
4. 与 checkpoint、logger 的交互点。

**输出物**：一页「调用序列图」（缩进列表或 mermaid），便于日后快速定位代码。

---

## 8. 建议的第一周打卡顺序

1. README + 官方 Parallelism 指南浏览（建概念，不抠完所有细节）。
2. 选定一个 `examples/`：跑通或至少读通入口参数。
3. Phase 1：`transformer` 单层 + 训练循环骨架。
4. Phase 2：先 **distributed/数据并行相关**，再 **TP**，再 **PP**；其余按目标模型按需追加。
